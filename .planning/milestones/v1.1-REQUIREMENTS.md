# Requirements: Haaabit

**Defined:** 2026-03-08
**Core Value:** Let AI accurately understand what the user needs to do today and reliably complete habit check-ins on the user's behalf.

## v1.1 Requirements

Requirements for the v1.1 `Impeccable UI Polish` milestone. Each requirement mapped to roadmap phases for this milestone.

### Design System Foundation

- [x] **DSGN-01**: User experiences one shared visual language across auth, dashboard, habits, detail, and API Access instead of page-by-page styling drift.
- [x] **DSGN-02**: User sees consistent typography, spacing, color, border, and elevation rules across all primary product surfaces.
- [x] **DSGN-03**: User interacts with consistent controls, forms, cards, badges, alerts, and overlays throughout the web app.

### Shell, States, and Responsiveness

- [x] **UXSH-01**: User can navigate a coherent signed-in shell with clear hierarchy, active navigation, and mobile-safe layout behavior.
- [x] **UXSH-02**: User sees polished loading, empty, error, success, and disabled states across all major web flows.
- [x] **UXSH-03**: User can use the app effectively on mobile and desktop with layouts that reprioritize content instead of shrinking desktop UI.
- [x] **UXSH-04**: User receives restrained motion and transition cues that improve continuity without adding visual noise.

### Auth Experience

- [x] **UXAU-01**: User can sign in or create an account through a trust-forward auth surface with clear hierarchy and strong field affordances.
- [x] **UXAU-02**: User receives actionable validation, pending, and error feedback during auth flows without losing entered context.

### Dashboard and Today

- [x] **UXTD-01**: User can scan today's priorities at a glance with Today clearly prioritized over supporting analytics.
- [x] **UXTD-02**: User receives polished completion, undo, and quantity-update feedback without losing page context.
- [x] **UXTD-03**: User can distinguish no-habits, nothing-due, all-done, loading, and error states on the dashboard.

### Habits and Detail

- [x] **UXHB-01**: User can manage habits through a coherent toolbar, readable cards, and consistent action placement across breakpoints.
- [x] **UXHB-02**: User can create, edit, archive, and restore habits through one consistent overlay/action language with accessible interaction behavior.
- [x] **UXHB-03**: User can view habit detail with summary-first hierarchy, readable trend/history sections, and preserved list context.

### API Access

- [x] **UXAP-01**: User can manage personal API tokens in a settings-grade surface with clear security, reveal, copy, and rotate guidance.
- [x] **UXAP-02**: User can find a clear first-call quickstart and docs/spec entry points from the API Access page.

### Quality and Hardening

- [x] **QUAL-01**: User can use the polished UI without regressions to existing auth, today, habits, stats, and API behavior.
- [x] **QUAL-02**: User can use polished flows with preserved keyboard/focus behavior, readable contrast, and reduced-motion-safe interactions.

## Future Requirements

Deferred beyond the v1.1 polish milestone.

### Visual Expansion

- **VISX-01**: User can switch to a fully supported dark theme with parity across all core surfaces.
- **VISX-02**: User can choose density, accent, or personalization options without fragmenting the product's visual language.
- **VISX-03**: User can use keyboard shortcuts or command-style navigation for faster movement across the app.

### AI and Platform Expansion

- **AINT-01**: AI client can use MCP tools that wrap the same domain capabilities as the REST API.
- **AINT-02**: AI client can retrieve richer interruption analysis and habit-risk summaries for coaching flows.
- **NOTF-01**: Operator can configure outbound email delivery for reminder-related workflows.
- **NOTF-02**: System can send reminder emails based on user-configured rules.

## Out of Scope

Explicit exclusions for the v1.1 milestone.

| Feature | Reason |
|---------|--------|
| New habit semantics or recurrence-rule expansion | This milestone is about presentation and interaction quality, not domain-model changes |
| Passkeys, OAuth providers, or auth-model expansion | Auth trust and usability should improve without widening backend/auth scope |
| Calendar, board, timeboxing, or alternate information architectures | These would multiply IA and responsive work before the default experience is excellent |
| Full dark-mode parity | Light mode is the deliberate first-class theme for v1.1 |
| Gamification layers such as confetti, badges, or points | Conflicts with the calm, refined, reliable product direction |
| New analytics concepts, reminders, or coaching surfaces | Adds product scope instead of fixing the shipped product surface |
