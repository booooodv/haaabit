# Requirements: Haaabit

**Defined:** 2026-03-07
**Core Value:** Let AI accurately understand what the user needs to do today and reliably complete habit check-ins on the user's behalf.

## v1 Requirements

Requirements for initial release. Each maps to roadmap phases.

### Authentication

- [x] **AUTH-01**: User can register an account with email and password.
- [x] **AUTH-02**: User can log in with email and password and remain authenticated across browser refresh.
- [x] **AUTH-03**: User can log out from the web app at any time.
- [x] **AUTH-04**: User can access only their own habits, check-ins, and statistics after authentication.

### Habits

- [x] **HABT-01**: User can create a habit with a name, description, category, target value, unit, start date, and active state.
- [x] **HABT-02**: User can define a habit frequency as daily, weekly N times, specific weekdays, or monthly N times.
- [x] **HABT-03**: User can edit an existing habit's details and future schedule.
- [x] **HABT-04**: User can archive a habit without deleting its historical records.
- [x] **HABT-05**: User can view a list of active and archived habits with search or basic filtering by state/category.

### Today View

- [x] **TODA-01**: User can view all habits due on the current day in a dedicated "today" view.
- [x] **TODA-02**: User can see today's habits separated into completed and pending groups.
- [x] **TODA-03**: User can see today's total habits, completed count, pending count, and completion rate in the same view.
- [x] **TODA-04**: AI client can retrieve a structured today summary containing date, total, completed, pending, and grouped habit lists.

### Check-Ins

- [x] **CHKI-01**: User can mark a due habit complete from the web app.
- [x] **CHKI-02**: User can undo a habit completion from the web app.
- [x] **CHKI-03**: User can record a quantified completion value for habits that use numeric targets.
- [x] **CHKI-04**: AI client can mark a habit complete through a REST API.
- [x] **CHKI-05**: AI client can undo a habit completion through a REST API.
- [x] **CHKI-06**: Each check-in stores its date, timestamps, completion value when provided, optional note, and mutation source.

### History And Statistics

- [x] **STAT-01**: User can view a habit detail page with current streak, longest streak, total completions, failure/interruption count, and recent history.
- [x] **STAT-02**: User can view account-level overview metrics including today's completion count, overall weekly completion rate, active habit count, and habit stability rankings.
- [x] **STAT-03**: User can view 7-day and 30-day trend data for habits or overall activity.
- [x] **STAT-04**: AI client can retrieve structured overview statistics and habit-level statistics through REST APIs.
- [x] **STAT-05**: Statistics remain consistent with check-in history after completion and undo actions.

### API

- [x] **API-01**: AI client can list habits through a REST API.
- [x] **API-02**: AI client can retrieve a habit's detail through a REST API.
- [x] **API-03**: AI client can create a new habit through a REST API.
- [x] **API-04**: AI client can update an existing habit's definition or frequency through a REST API.
- [x] **API-05**: REST APIs return stable JSON field names and explicit error responses.
- [x] **API-06**: The project exposes machine-readable API documentation for the supported REST endpoints.

### Self-Hosting

- [x] **SELF-01**: Operator can start the app and required services with Docker Compose using documented steps.
- [x] **SELF-02**: Operator can configure the app with a documented `.env.example`.
- [x] **SELF-03**: Operator can run database migrations during first setup and upgrades using documented commands.
- [x] **SELF-04**: Operator can verify service health through a documented health check endpoint or command.

## v2 Requirements

Deferred to future release. Tracked but not in current roadmap.

### AI Integrations

- **AINT-01**: AI client can use MCP tools that wrap the same domain capabilities as the REST API.
- **AINT-02**: AI client can retrieve richer interruption analysis and habit-risk summaries for coaching flows.

### Notifications

- **NOTF-01**: Operator can configure outbound email delivery for reminder-related workflows.
- **NOTF-02**: System can send reminder emails based on user-configured rules.

### Product Expansion

- **PROD-01**: User can access a dedicated API/MCP debug page in the web app.
- **PROD-02**: User can compare multiple habits through richer analytics views and visualizations.
- **PROD-03**: Multiple users can collaborate or share habit spaces with role-based permissions.

## Out of Scope

Explicitly excluded. Documented to prevent scope creep.

| Feature | Reason |
|---------|--------|
| Built-in AI reminder loop orchestration | External AI systems own the reminder cycle in the chosen product direction |
| Gamification systems (badges, points, leaderboards) | Does not strengthen the core AI-first habit workflow in v1 |
| Real-time sync / websocket updates | Adds infrastructure complexity without being necessary for the initial personal-use experience |
| Arbitrary custom recurrence builder | High edge-case cost; v1 should stick to defined daily/weekly/monthly patterns |
