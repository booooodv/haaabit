# Milestone v1.2: Chinese and English Localization

**Status:** ✅ SHIPPED 2026-03-08
**Phases:** 12-14
**Total Plans:** 8

## Overview

Add calm, dependable bilingual support so the shipped product can be used in both Chinese and English without changing core habit semantics or AI behavior.

## Phases

### Phase 12: Localization Foundation and Locale Switching

**Goal**: Users land in Chinese or English automatically, can switch languages explicitly, and keep that language preference on later visits.
**Depends on**: Phase 11 (v1.1 archived baseline)
**Plans**: 2 plans

Plans:

- [x] 12-01: Establish shared locale detection, fallback, and persistence
- [x] 12-02: Add in-place locale switching across auth and shell

**Details:**
- Requirements: `I18N-01`, `I18N-02`, `I18N-03`, `I18N-04`
- Success Criteria:
  1. User sees auth and shared shell chrome load in one chosen language instead of mixed Chinese and English strings.
  2. User lands in Chinese or English based on browser language on first visit, with a predictable fallback when the browser locale is unsupported.
  3. User can switch languages from the UI without being bounced off the current page or losing in-progress context.
  4. User returns to the previously selected language on later visits after manually switching.

### Phase 13: Core Product Surface Localization

**Goal**: Users can operate the main shipped product flows in Chinese or English with localized UI states and unchanged habit data.
**Depends on**: Phase 12
**Plans**: 3 plans

Plans:

- [x] 13-01: Localize dashboard and today surfaces
- [x] 13-02: Localize habits management and detail surfaces
- [x] 13-03: Localize API Access and close product-surface regression

**Details:**
- Requirements: `L10N-01`, `L10N-02`, `L10N-03`
- Success Criteria:
  1. User sees localized navigation, labels, forms, and helper copy across auth, dashboard, habits, detail, and API Access.
  2. User sees localized loading, empty, success, warning, and error states without mixed-language fallback in supported surfaces.
  3. User's self-entered habit names, categories, and numbers remain exactly as entered while surrounding product chrome changes language.

### Phase 14: Docs, Verification, and Localization Hardening

**Goal**: Users and operators can understand the product and platform in both languages, and the bilingual release holds up under regression and layout checks.
**Depends on**: Phase 13
**Plans**: 3 plans

Plans:

- [x] 14-01: Localize quickstart and self-host documentation
- [x] 14-02: Localize `/api/docs` while preserving the English API contract
- [x] 14-03: Close bilingual regression, accessibility, and docs integrity gates

**Details:**
- Requirements: `DOCS-01`, `DOCS-02`, `DOCS-03`, `QLOC-01`, `QLOC-02`
- Success Criteria:
  1. User can read core help, self-host guidance, API docs, and quickstart content in Chinese and English.
  2. Operator can understand locale detection, manual switching, and fallback behavior from the docs.
  3. User can complete the main product flows in both languages without untranslated copy or layout-breaking text in supported surfaces.
  4. User can switch language on desktop and mobile without breaking navigation state, focus continuity, or reduced-motion-safe behavior.

## Milestone Summary

**Decimal Phases:**

- None

**Key Decisions:**

- Decision: Limit v1.2 to Simplified Chinese and English. Outcome: ✓ Good
- Decision: Keep one shared route structure instead of adding locale-prefixed app routes. Outcome: ✓ Good
- Decision: Preserve user-authored habit data and API payload contracts exactly as entered. Outcome: ✓ Good
- Decision: Keep technical literals such as commands, paths, headers, schema keys, and bearer syntax in English. Outcome: ✓ Good
- Decision: Let `/api/docs` localize its UI while `/api/openapi.json` remains English-only. Outcome: ✓ Good

**Issues Resolved:**

- Removed fixed English product-owned copy from shipped auth, dashboard, habits, detail, and API Access surfaces.
- Added bilingual repository docs and locale-behavior operator guidance.
- Removed stale absolute worktree links from `README.md` and self-host docs.
- Added release-gate regression coverage for desktop/mobile locale switching, focus continuity, reduced motion, and API docs access.

**Issues Deferred:**

- Additional locales beyond Simplified Chinese and English remain deferred.
- Account-level synced locale preference remains deferred.
- API payload localization remains deferred.
- Reminder-email workflows and broader AI-side integrations remain future scope.

**Technical Debt Incurred:**

- Phase 12 shipped with passing verification but without a `12-VALIDATION.md` Nyquist artifact; this was accepted at milestone archive time as process debt rather than product debt.
