# Requirements: Haaabit

**Defined:** 2026-03-09
**Core Value:** Let AI accurately understand what the user needs to do today and reliably complete habit check-ins on the user's behalf.

## v1.5 Requirements

Requirements for the v1.5 `MCP Integration` milestone.

### MCP Foundation

- [x] **MCP-01**: Operator can run a standalone Haaabit MCP server from the monorepo and from a published npm package without starting a second Haaabit-specific backend service.
- [x] **MCP-02**: MCP server starts over local `stdio`, declares tool capability metadata correctly, and is discoverable by generic MCP clients.
- [x] **MCP-03**: MCP server reads `HAAABIT_API_URL` and personal API token style credentials from configuration/environment, fails clearly when config is missing or invalid, and never prints raw secrets in logs or tool outputs.
- [x] **MCP-04**: MCP tool definitions reuse existing shared contracts and route semantics so Haaabit REST changes do not require a second hand-maintained schema surface.

### Read Tools

- [x] **READ-01**: AI client can list habits through MCP with the same filter surface exposed by `GET /api/habits`.
- [x] **READ-02**: AI client can fetch full habit detail through MCP with the same summary stats, history, and trend data exposed by `GET /api/habits/:habitId`.
- [x] **READ-03**: AI client can fetch the canonical today summary through MCP with the same pending/completed breakdown exposed by `GET /api/today`.
- [x] **READ-04**: AI client can fetch overview analytics through MCP with the same metrics, trends, and stability ranking exposed by `GET /api/stats/overview`.

### Mutation Tools

- [x] **WRITE-01**: AI client can create a habit through MCP with the same validations and saved resource shape exposed by `POST /api/habits`.
- [x] **WRITE-02**: AI client can update a habit through MCP with the same editable fields, validations, and saved resource shape exposed by `PATCH /api/habits/:habitId`.
- [x] **WRITE-03**: AI client can archive and restore habits through MCP with the same state-transition semantics exposed by the REST API.
- [x] **WRITE-04**: AI client can complete boolean today actions, set quantified totals, and undo today actions through MCP with the same canonical summary refresh behavior exposed by the REST API.

### Error Fidelity and Release Shape

- [x] **ERR-01**: MCP tools surface REST auth, validation, not-found, and archived-habit conflicts with actionable MCP-facing failures that preserve Haaabit semantics without leaking secrets.
- [x] **REL-01**: Published package includes built artifacts, CLI/bin wiring, README setup steps, and generic-client-friendly npm/npx configuration examples.
- [x] **REL-02**: v1.5 includes verification coverage for tool registration, schema alignment, config validation, REST error mapping, and at least one end-to-end MCP interaction path against the existing Haaabit API.

## Future Requirements

### MCP Expansion

- **MCPX-01**: Haaabit MCP server can also run as a remote Streamable HTTP server for hosted or shared deployments.
- **MCPX-02**: Haaabit MCP package exposes resources and/or prompts where they improve client UX beyond the core tool surface.
- **MCPX-03**: Package includes official MCP Registry publication metadata and verified namespace ownership.

### Product Expansion

- **NOTF-01**: Operator can configure outbound email delivery for reminder-related workflows.
- **NOTF-02**: System can send reminder emails based on user-configured rules.
- **VISX-01**: User can switch to a fully supported dark theme with parity across all core surfaces.
- **PROD-01**: User can move faster through the app with deeper keyboard-first productivity features.

## Out of Scope

| Feature | Reason |
|---------|--------|
| Browser session, sign-up, or admin registration routes as MCP tools | v1.5 is intentionally scoped to the personal-token-compatible business API surface |
| New domain capabilities that do not already exist in the REST API | This milestone is an adapter/integration milestone, not a product-semantic expansion |
| Remote Streamable HTTP transport | Valuable, but it adds extra protocol and auth complexity beyond the chosen local `stdio` deliverable |
| Token issuance/rotation from within MCP | The MCP package should consume an existing personal token, not bootstrap or rotate its own credentials |
| Optional publication polish on the web app (`error.tsx`, `not-found.tsx`, Docker `pnpm` parity) | Deferred so v1.5 stays focused on MCP delivery |
