# Milestone v1.5: MCP Integration

**Status:** ✅ SHIPPED 2026-03-09
**Phases:** 18-21
**Total Plans:** 11

## Overview

v1.5 added a standalone MCP package inside the monorepo without creating a second backend or a second auth model. The milestone shipped a real `@haaabit/mcp` package, full personal-token-compatible habits/today/stats tool coverage, package-local operator docs, and a release gate that proves the built stdio server against the existing Haaabit API.

## Phases

### Phase 18: MCP Package Foundation and Contract Mapping

**Goal**: Create the `packages/mcp` foundation, lock scope boundaries, and establish one contract-aware path from MCP tools to the existing API.
**Depends on**: Phase 17 and the shipped v1.4 codebase
**Plans**: 3 plans

Plans:

- [x] 18-01: Scaffold `packages/mcp` package, build config, CLI entrypoint, and startup configuration validation
- [x] 18-02: Build shared REST client and MCP error-mapping layer against the existing bearer-authenticated API
- [x] 18-03: Define tool inventory, naming, and contract adapters for the full personal-token-compatible API surface

**Details:**
- Requirements: `MCP-01`, `MCP-02`, `MCP-03`, `MCP-04`
- Outcomes: package skeleton, `stdio` bootstrap, env validation, shared API client, explicit scope boundaries, and parity map between API routes and MCP tools.
- Success criteria:
  1. A generic MCP client can start the Haaabit MCP process over `stdio` without custom manual boot steps.
  2. Missing or invalid API configuration fails with actionable messages and without echoing secrets.
  3. Every promised habits/today/stats route is represented in one explicit MCP tool inventory.
  4. Shared contract adapters exist so tool schemas are not handwritten ad hoc per operation.

### Phase 19: Read Tool Surface

**Goal**: Ship the read-only MCP tools that expose the existing habits, today, and stats views in a structured, generic-client-friendly way.
**Depends on**: Phase 18
**Plans**: 3 plans

Plans:

- [x] 19-01: Implement habits list and habit detail tools
- [x] 19-02: Implement today summary and stats overview tools
- [x] 19-03: Add verification for read-tool registration, schemas, and structured outputs

**Details:**
- Requirements: `READ-01`, `READ-02`, `READ-03`, `READ-04`
- Outcomes: discoverable read tools with structured content, stable names/descriptions, and parity with existing GET routes.
- Success criteria:
  1. MCP clients can discover read tools with clear descriptions and validated input schemas.
  2. Habit list/detail tools return the same domain data the REST API returns for equivalent requests.
  3. Today summary and stats overview tools expose canonical read models instead of degraded text-only summaries.
  4. Read-path verification proves tool outputs stay aligned with shared contracts.

### Phase 20: Mutation Tool Surface and Error Semantics

**Goal**: Add write tools for habits and today actions while preserving existing validation, conflict, and canonical refresh behavior.
**Depends on**: Phase 19
**Plans**: 3 plans

Plans:

- [x] 20-01: Implement habit create and update tools
- [x] 20-02: Implement archive/restore and today mutation tools
- [x] 20-03: Verify mutation-path error handling, conflict semantics, and canonical response behavior

**Details:**
- Requirements: `WRITE-01`, `WRITE-02`, `WRITE-03`, `WRITE-04`, `ERR-01`
- Outcomes: full write parity for personal-token-compatible routes and reliable MCP-facing failures for bad auth/input/state.
- Success criteria:
  1. MCP clients can create, update, archive, and restore habits through the same validations and resource semantics as REST.
  2. MCP clients can complete, set total, and undo today actions and receive refreshed canonical summaries.
  3. Archived-habit conflicts, auth failures, validation failures, and not-found paths are surfaced consistently and actionably.
  4. Mutation tools do not fork habit logic or introduce state behavior absent from the existing API.

## Milestone Summary

**Decimal Phases:**

- None

**Key Decisions:**

- Decision: v1.5 MCP should live in `packages/mcp` and stay a thin adapter over the existing REST API. Outcome: ✓ Good
- Decision: The milestone should cover the full personal-token-compatible habits, today, and stats API surface, not browser-session/admin flows. Outcome: ✓ Good
- Decision: Local `stdio` delivery for generic MCP clients comes before any remote HTTP transport work. Outcome: ✓ Good
- Decision: Package/docs/publish shape is part of the milestone, not cleanup after the code works. Outcome: ✓ Good

**Issues Resolved:**

- Added a real standalone MCP package inside the monorepo instead of keeping MCP as planning-only scope.
- Reused shared contracts and existing API semantics so MCP no longer needs hand-maintained duplicate schemas.
- Shipped real read and write MCP tools for the personal-token-compatible Haaabit API surface.
- Added package-local docs and a release gate that prove the built stdio server against the existing API.

**Issues Deferred:**

- Remote Streamable HTTP transport.
- MCP registry publication metadata.
- Web-publication polish leftovers such as `error.tsx`, `not-found.tsx`, and Docker `pnpm` version parity.

**Technical Debt Incurred:**

- No new product-level technical debt beyond the intentional deferrals above.
- Phase 20 and Phase 21 verification artifacts were reconstructed retroactively during milestone closeout so the audit chain is now complete.
