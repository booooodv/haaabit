# Requirements: Haaabit

**Defined:** 2026-03-08
**Core Value:** Let AI accurately understand what the user needs to do today and reliably complete habit check-ins on the user's behalf.

## v1.2 Requirements

Requirements for the v1.2 `Chinese and English Localization` milestone.

### Localization Foundation

- [x] **I18N-01**: User can open the product in Chinese or English through one shared locale system instead of mixed hard-coded copy.
- [x] **I18N-02**: User's first visit defaults to Chinese or English from browser language, with a safe fallback when the browser locale is unsupported.
- [x] **I18N-03**: User can switch between Chinese and English from the product UI without losing current page context.
- [x] **I18N-04**: User returns to the same chosen language on later visits after manually switching.

### Product Surface Localization

- [x] **L10N-01**: User sees localized navigation, labels, buttons, forms, helper text, and validation copy across auth, dashboard, habits, detail, and API Access.
- [x] **L10N-02**: User sees localized loading, empty, success, warning, and error states across the main web flows without mixed-language fallback.
- [x] **L10N-03**: User sees self-entered habit names, categories, and numeric data preserved as entered while surrounding product chrome changes language.

### Docs and Developer Surface

- [x] **DOCS-01**: User can read core product help and self-host instructions in both Chinese and English.
- [x] **DOCS-02**: API consumer can use the in-product API docs and quickstart guidance in both Chinese and English.
- [x] **DOCS-03**: Operator can understand locale detection, manual switching, and fallback behavior from the docs.

### Quality and Hardening

- [x] **QLOC-01**: User can complete the main auth, dashboard, habits, detail, and API Access flows in both languages without untranslated or layout-breaking copy in the supported surfaces.
- [x] **QLOC-02**: User can change language on desktop and mobile without breaking navigation state, focus continuity, or reduced-motion-safe behavior.

## Outcomes

- The shared locale system shipped with browser-language defaulting, English fallback, and remembered manual switching.
- Product-owned copy shipped bilingually across auth, dashboard, today, habits, detail, API Access, repository docs, and `/api/docs`.
- User-authored habit names, categories, units, and numeric values remained unchanged across locale changes.
- `/api/openapi.json` stayed English-only while `/api/docs` localized only its explanatory UI.

## Future Requirements

### Localization Expansion

- **I18N-05**: User can keep the same language preference at the account level across devices after signing in.
- **I18N-06**: User can use the product in more than Chinese and English without restructuring the locale system.
- **I18N-07**: Operator can localize API success/error payloads for human-facing API consumers without forking AI contracts.

### Product Expansion

- **VISX-01**: User can switch to a fully supported dark theme with parity across all core surfaces.
- **AINT-01**: AI client can use MCP tools that wrap the same domain capabilities as the REST API.
- **NOTF-01**: Operator can configure outbound email delivery for reminder-related workflows.
- **NOTF-02**: System can send reminder emails based on user-configured rules.

## Out of Scope

| Feature | Reason |
|---------|--------|
| Automatic translation of user-entered habit names, categories, or notes | The milestone localized product-owned copy only and did not mutate user-authored data |
| API response and error payload localization | Existing REST contracts stayed stable for AI and integration clients |
| More than Chinese and English locales | The milestone was intentionally scoped to two supported languages |
| Account-level synced language preference across devices | Browser default plus remembered local override was sufficient for this pass |
| New habit semantics, reminders, or AI capability expansion | Localization stayed focused on usability rather than widening domain scope |
