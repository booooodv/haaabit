# Roadmap: Haaabit

## Overview

Haaabit will be built as an AI-first, self-hosted habit tracking product in six phases. The roadmap starts by locking down authentication and the core habit model, then makes "today" and check-in behavior trustworthy, expands into habit management and analytics, exposes the broader AI-facing REST platform, and finishes by hardening the self-hosted delivery path for real users.

## Phases

**Phase Numbering:**
- Integer phases (1, 2, 3): Planned milestone work
- Decimal phases (2.1, 2.2): Urgent insertions (marked with INSERTED)

Decimal phases appear between their surrounding integers in numeric order.

- [x] **Phase 1: Authentication and Core Habit Model** - Establish secure access and the supported habit/schedule semantics.
- [x] **Phase 2: Today Engine and Check-In Flows** - Deliver the core "what should I do today?" and reversible completion logic.
- [x] **Phase 3: Habit Management and Detail Views** - Let users manage habits safely and inspect per-habit history.
- [x] **Phase 4: Analytics and Overview** - Turn check-in history into trustworthy trends and summary metrics.
- [x] **Phase 5: AI REST Platform and Contracts** - Broaden the AI-facing API surface and document it clearly.
- [x] **Phase 6: Self-Hosted Delivery Hardening** - Make setup, migration, and health verification reliable for operators.

## Phase Details

### Phase 1: Authentication and Core Habit Model
**Goal**: Deliver secure account access and the core data model for users and scheduled habits.
**Depends on**: Nothing (first phase)
**Requirements**: AUTH-01, AUTH-02, AUTH-03, AUTH-04, HABT-01, HABT-02
**Success Criteria** (what must be TRUE):
  1. User can register, log in, stay logged in across refresh, and log out successfully.
  2. Authenticated users can only access their own habit data.
  3. User can create a habit with the supported fields and supported daily/weekly/monthly schedule types.
**Plans**: 3 plans

Plans:
- [x] 01-01: Set up repository structure, auth foundation, and user persistence
- [x] 01-02: Model habits and supported recurrence rules in the database and domain layer
- [x] 01-03: Deliver account flows and initial habit creation UI/API path

### Phase 2: Today Engine and Check-In Flows
**Goal**: Make today's due habits, completion, undo, and AI task-oriented actions correct and trustworthy.
**Depends on**: Phase 1
**Requirements**: TODA-01, TODA-02, TODA-03, TODA-04, CHKI-01, CHKI-02, CHKI-03, CHKI-04, CHKI-05, CHKI-06
**Success Criteria** (what must be TRUE):
  1. User sees today's habits separated into completed and pending groups with accurate summary counts.
  2. User can complete and undo habits, including quantified progress where applicable.
  3. AI client can fetch a structured today summary and perform complete/undo actions while the system records mutation source.
**Plans**: 3 plans

Plans:
- [x] 02-01: Build and test the recurrence evaluation and today-summary domain services
- [x] 02-02: Implement check-in persistence, quantified completion, undo, and provenance rules
- [x] 02-03: Ship today dashboard flow plus task-oriented AI endpoints for today/complete/undo

### Phase 3: Habit Management and Detail Views
**Goal**: Complete the core user workflow for maintaining habits and inspecting individual habit history.
**Depends on**: Phase 2
**Requirements**: HABT-03, HABT-04, HABT-05, STAT-01
**Success Criteria** (what must be TRUE):
  1. User can edit, archive, search, and filter habits without losing history.
  2. User can open a habit detail view showing streaks, totals, interruption count, and recent records.
  3. Habit edits and archive state changes do not corrupt existing history or break the today flow.
**Plans**: 3 plans

Plans:
- [x] 03-01: Build habit list management, search/filter, edit, and archive flows
- [x] 03-02: Create habit detail read models and recent-history presentation
- [x] 03-03: Add regression coverage around edits, archives, and history integrity

### Phase 4: Analytics and Overview
**Goal**: Provide trustworthy overview metrics and trend views for both users and AI clients.
**Depends on**: Phase 3
**Requirements**: STAT-02, STAT-03, STAT-04, STAT-05
**Success Criteria** (what must be TRUE):
  1. User can view account-level overview metrics and 7-day/30-day trends.
  2. AI client can retrieve overview and habit-level statistics through structured REST responses.
  3. Statistics remain consistent with underlying history after completion and undo actions.
**Plans**: 3 plans

Plans:
- [x] 04-01: Implement canonical stats calculation and aggregation queries
- [x] 04-02: Deliver overview/trend endpoints and dashboard analytics views
- [x] 04-03: Verify stats consistency against history and mutation edge cases

### Phase 5: AI REST Platform and Contracts
**Goal**: Expand the REST API into a complete, documented integration surface for AI assistants.
**Depends on**: Phase 4
**Requirements**: API-01, API-02, API-03, API-04, API-05, API-06
**Success Criteria** (what must be TRUE):
  1. AI client can list, detail, create, and update habits through stable REST endpoints.
  2. API responses use stable JSON contracts with explicit validation and error behavior.
  3. Developers can discover supported endpoints through machine-readable API documentation.
**Plans**: 2 plans

Plans:
- [x] 05-01: Broaden the REST surface for habit listing/detail/create/update workflows
- [x] 05-02: Add contract validation, error model consistency, and generated API documentation

### Phase 6: Self-Hosted Delivery Hardening
**Goal**: Make fresh deployment, upgrades, and health verification dependable for self-hosted operators.
**Depends on**: Phase 5
**Requirements**: SELF-01, SELF-02, SELF-03, SELF-04
**Success Criteria** (what must be TRUE):
  1. Operator can start the full stack with Docker Compose using documented steps.
  2. Operator can configure the app from `.env.example`, run migrations, and verify health.
  3. A clean-clone install works without tribal knowledge and has an upgrade path documented.
**Plans**: 2 plans

Plans:
- [x] 06-01: Deliver Docker Compose, environment template, migrations flow, and health checks
- [x] 06-02: Write and verify self-hosted setup and upgrade documentation on a clean install path
